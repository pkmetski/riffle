# Chitanka Source and server-side facets

We add [chitanka.info](https://chitanka.info) (EPUB ebooks) and [gramofonche.chitanka.info](https://gramofonche.chitanka.info) (MP3 audiobooks) as a third shipping [Source] type (`SourceType.CHITANKA`), following [ADR 0041]. Both sites are anonymous public catalogues in Bulgarian, served by one HTML-scraping [Catalog] implementation that exposes them as two [Libraries] (Books, Audiobooks) within a single Source.

The two sites' navigation shape does not fit Riffle's flat-Library-per-root model — Chitanka alone carries 40k+ books navigated through categories, genres, and series. To handle this without cramming a category tree into the [Navigation Drawer], we grow the [Catalog] interface with a **server-side facet** mechanism and render the site's categories as a chip strip inside the Library screen. This is deliberately distinct from ADR 0027's client-side facets (which filter already-synced items in memory): a server-side facet is a hint passed back to `Catalog.browse(rootId, facet=…)` that the Source executes against its origin. `Catalog` grows two members: `listFacets(rootId)` and an optional `facet: FacetSelection?` parameter on `browse`. Sources without server-side facets return `emptyList()` and don't consume the parameter — the strip simply doesn't render. This is an abstraction-growth event, not a Chitanka-specific special case: any future Source that pushes filters server-side (OPDS categories, Komga genres) uses the same shape.

## Corollaries

- **One Chitanka Source per user, zero-config.** No credentials, no server URL — both domains are hardcoded. Add-Source is a toggle; a second attempt returns the existing row. On install, the Catalog does a `connectivityCheck` and eagerly warms two Room-cached indexes: the top-level category list (30–40 entries from `/books/category` + `/collections/school` + `/collections/university`) and the flat series list (~2,300 entries enumerated across `/series/alpha/{letter}` for the Cyrillic alphabet). Both refresh on a 7-day TTL.
- **`sourceItemId` is the URL path.** E.g. `text/44723-abu-hasan`, `prikazki/abanosoviia-kon`. Chitanka pages that live at both `/book/M-slug` and `/text/N-slug` (same book, different surfaces) canonicalise to `/text/N-slug` at scrape time — that's the surface carrying the EPUB link and genre labels. Trade-off: a slug rename orphans downloaded items; the escape hatch is the same as everywhere else in Riffle, re-add.
- **Browse pages are Room-cached with a 24 h TTL**, keyed by `(sourceId, libraryId, facetKey, page)`. Detail pages (`getItem`) same. This is a **performance** cache — a page-cache, not a full-catalog mirror — and as a side effect makes previously-visited pages available offline. `ChitankaCatalog` claims `OfflineBrowseCapability` on that basis: uncached pages surface a "no connection" state, cached pages serve normally.
- **No `catalog_cache` full sync.** Chitanka is an unbounded catalogue in the sense of [ADR 0041] — syncing 40k rows into Room is wrong on write cost, wrong on staleness, and unnecessary for the pagination-driven browse UX. The 24 h page-cache is the entire persistence story.
- **No explicit "add to library" verb.** Riffle records a chitanka item in local tables **on first interaction** — opening the detail screen, streaming a Gramofonche audiobook, downloading the EPUB. Progress persists keyed on `(sourceId, sourceItemId)` from that first touch. There is no wishlist, no "My Books" tab distinct from browse; downloaded bytes live on the standard [Downloads Screen].
- **Gramofonche audiobooks stream directly from the site.** `AudiobookMediaCapability.buildStreamUrl` returns the raw `.mp3` URL (percent-encoded via `URI` for Cyrillic filenames); ExoPlayer range-fetches it the same way it does for [ABS]. Tracks come out of `getTracks` as one entry per MP3 in page order. No chapter markers are synthesised — `getAudiobookChapters` returns empty; chapter navigation degrades to track navigation, which is sufficient because Gramofonche listeners are not reading along. Items with only a ZIP download and no per-track MP3s are marked unplayable; on-device zip-extraction for audio playback is out of scope for v1.
- **`SeriesCapability` yes, `CollectionsCapability` no.** Chitanka's `/series/alpha/{letter}` supports enumeration; `listSeries("books")` serves from the warmed cache; `listSeries("audiobooks")` is empty. Chitanka's two editorial "collections" (`/collections/school`, `/collections/university`) are not user-created bundles and are the wrong semantic for Riffle's Collections tab — they surface instead as two extra chips in the Books Library's [Category Facet] strip alongside the genre chips.
- **Parsing is isolated in a dedicated `core:catalog-chitanka` Gradle module** depending only on `core:catalog` (interfaces), jsoup for HTML parsing, and Riffle's HTTP primitives. jsoup, `<article class="book-media">` selectors, and every Cyrillic URL literal live only inside that module. Everything the rest of the app touches is Catalog vocabulary. Deleting the module + one Hilt binding removes the Source entirely.
- **429 backoff and User-Agent identification.** The scraper honours a 3-attempt retry with 1.5 s / 3 s waits on HTTP 429 and identifies itself with `Riffle/<version> (Android) chitanka-source` so the upstream can throttle or block cleanly if it needs to.
- **Bulgarian collation** (`Collator.getInstance(Locale("bg"))`) is used for sorting titles, authors, categories, and series names inside the Source; the rest of Riffle is unaffected.
- **Content-source disclosure.** The Add-Source install screen carries a one-line notice: "Съдържанието се предоставя от chitanka.info и gramofonche.chitanka.info" (Content is provided by chitanka.info and gramofonche.chitanka.info). Riffle scrapes only public HTML — no login, no cookie theft, no `robots.txt` bypass.

## Scope of the first change

Ship `SourceType.CHITANKA`, the `core:catalog-chitanka` module (jsoup port of the reference [chitanka-to-audiobookshelf](https://github.com/pkmetski/chitanka-to-audiobookshelf) scrapers), the interface growth (`listFacets`, `browse(facet=…)`), the category-chip UI on the Library screen (gated on non-empty `listFacets`), the Add-Source toggle, and the Room tables backing the 7-day category+series index cache and the 24 h page cache. `ProgressPeerCapability`, `ReadingSessionsCapability`, `StatsCapability`, `PlaylistsCapability`, `BookmarksCapability` — all absent by nature of the upstream.

## Amendments to existing ADRs

- **[ADR 0027]** (Client-side facet filtering): unchanged in scope; the new server-side facet mechanism is a peer, not a replacement — client-side facets continue to serve "unread" and the metadata drill-ins on the [Library Item Detail Screen] for [ABS] and [LocalFiles].
- **[ADR 0041]** (Source and Service abstractions): amended — `SourceType` grows a third value (`CHITANKA`); `Catalog` grows `listFacets` and the `facet` parameter on `browse`; the "unbounded catalog = no offline browse" observation is nuanced by the page-cache mechanism, which lets an unbounded Catalog still claim `OfflineBrowseCapability` for the parts it has cached.
