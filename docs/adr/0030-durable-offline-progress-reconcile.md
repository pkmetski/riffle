# ADR 0030 — Durable offline progress reconcile (dirty-set sweep + unified reconcile primitive)

**Status:** Accepted
**Extends:** [ADR 0008](0008-last-update-wins-progress-sync.md) — whose "if the server is unreachable the cycle is skipped; `localUpdatedAt` accumulates locally and is pushed on the next successful cycle" silently assumed the book would be **reopened**. It isn't always. This ADR makes the push durable and book-independent.
**Relates to:** [ADR 0019](0019-three-peer-unified-canonical-progress-sync.md), [ADR 0029](0029-audiobook-direct-abs-streaming-audio-led-sync.md). **Builds on** the local-position-persistence work now in `main` (schema v33): the `PositionStore<P>` / `TimestampedPositionStore<P>` abstraction, `ReadingPositionStore` (ebook, `reading_positions`) and `AudiobookPositionStore` (audio, `audiobook_positions`, `MIGRATION_32_33`), and `AudiobookPositionReconciler`.

## Context

Progress sync (ADR 0008/0019/0029) is already GET-before-PATCH last-update-wins, so it never overwrites a genuinely-newer server position — **while the open book's ~30s loop is running.** That loop only reconciles the *currently open* book. Progress made offline (downloaded ebook **or** downloaded audiobook — offline audiobook download shipped in #136, so ADR 0029's "out of v1 scope" line is stale) is held in the durable local position stores (`reading_positions`, `audiobook_positions`; both keyed `(serverId, itemId)` with a `localUpdatedAt`, schema v33) but is only pushed the next time *that same book* is reopened. A book that is never reopened never syncs — silent data loss. The v33 stores make the local position durable and able to *win on reopen*, but still push only via the open surface's loop; they do not close the never-reopened gap.

The requirement: queue offline progress and push it when connectivity returns **without** requiring a reopen, and **without** ever overwriting a server position that moved (and is newer) in the meantime.

## Decision

A **durable, book-independent reconcile** layered on top of the existing per-`(serverId, itemId)` position stores.

- **Coalesced per-target dirty set, not an event log.** A row is dirty when `localUpdatedAt > lastSyncedAt`. A new `lastSyncedAt: Long` column is added to **both** `reading_positions` and `audiobook_positions` (**migration 33→34**, on top of the current v33 baseline; default `0` ⇒ existing rows reconcile once, safe because reconcile is GET-before-PATCH). Successive offline edits coalesce into one row → one push.
- **`WorkManager`-backed sweep** (`NetworkType.CONNECTED`, `enqueueUniqueWork`) is the durable backbone, enqueued whenever a row goes dirty, plus a foreground kick and a periodic safety net. The ~30s in-reader/in-player live loop **stays** — it alone does the inbound *reader jump* and the responsive push for the open book, which a headless worker structurally cannot, and WorkManager's 15-minute floor is too coarse for it.
- **The dirty flag is the durability primitive; an explicit open-book exclusion is the safety primitive.** The dirty flag (`localUpdatedAt > lastSyncedAt`) decides *what the sweep enumerates*. Coordination with the open book, however, needs more than the dirty flag: the reader's `save()` re-stamps `localUpdatedAt = now` on every page turn, so an actively-read book is *continuously* dirty and the worker would not skip it. Worse, if the sweep reconciled the open book it could absorb a **cross-device server-win** into the row that the visible reader hasn't jumped to — leaving the reader behind the server, so the next page-turn overwrites the newer server position (a cross-device erase). The open book's own ~30s cycle is the only thing that can correctly apply such an inbound win (it has a reader to jump). Therefore a live surface registers its (serverId, itemId) — both ids for a matched book — in `OpenReconcileTargets` while open, and the sweep **skips** those targets. Durability is unaffected: the live cycle pushes the book while open, and its dirty row is reconciled by the next sweep once it closes. *(This corrects the original framing that the dirty flag alone made an open-book guard unnecessary.)*
- **Single shared per-target reconcile primitive** (Layer 1): capture `localUpdatedAt` → GET the ABS record → if server newer, **persist the server position locally** (not just the timestamp — the ADR 0008 reader path stamped only the timestamp because a live reader saved the position; a headless worker has no reader, so it must write the position) and set `localUpdatedAt = lastSyncedAt = serverStamp`; if local newer, PATCH and on success set `lastSyncedAt = localUpdatedAt` **iff `localUpdatedAt` did not advance mid-flight** (compare-and-clear), adopting the server's returned stamp as today; on any network failure, leave dirty and write nothing. Instantiated twice (ebook `String`/CFI, audio `Double`/seconds).
- **Layer-2 `CanonicalSyncCycle` is retained and refactored to compose Layer 1.** Matched-book live reading still needs multi-peer orchestration; its `SyncRemote.tryGet/tryPatch` become wrappers over the primitive. The primitive returns a **decision** and owns the *store* writes; the **caller** owns the consequence — the live reader/player jumps the visible surface (keeping `pendingServerJumpStamp`, the resume-at-reconciled guard, and the `localUpdatedAt = 0L` pre-settle transient guard from the erase saga), while the worker just persists.
- **The worker is single-target and multi-server.** It pushes whatever rows are dirty, one ABS record per row, and **never translates** — it never needs the bundle/cross-EPUB index. It enumerates dirty rows across **all** servers (not `getActive()`), groups by `serverId`, resolves each token, and skips servers with no valid token (their rows stay dirty).
- **A matched book is one position in two representations; reading and listening are the same activity, and the live matched path dual-writes both local stores.** Listening to a matched audiobook makes you genuinely current in the *ebook* coordinate too, and reading a matched ebook makes you current in the *audio* coordinate — so on every genuine position change the live matched path translates the canonical position through the bundle (SMIL + cross-EPUB index, already present in that path) and writes **both** `reading_positions` and `audiobook_positions`. Both rows are **first-class** position updates stamped `now()`, each following the identical lifecycle independently: on a successful live push, adopt *that* record's server stamp and mark clean; offline, stay dirty. The single-target worker then pushes whichever rows are dirty — translation-free, because the counterpart coordinate was already computed and stored at activity time. This adds a sibling-store dependency to each live surface (`EpubReaderViewModel` → `AudiobookPositionStore`, `AudiobookPlayerViewModel` → `ReadingPositionStore`). When a matched book's prerequisites are not cached (no bundle / no cross-EPUB index), translation is impossible, so the live path writes only the native store and degrades to single-peer exactly as the reconciliation cycle already does — the sibling row is then reconciled on the next open after the prerequisites land.
- **Only settled positions are persisted.** The dual-write obeys the existing erase-saga value guards (the player's `localUpdatedAt = 0L` / settle-epsilon for pre-seek/buffering ticks; the reader's resume-at-reconciled): a transient book-start/buffering position is never written to either store, so stamping `now()` is always a real position.

## Invariant

> The two ABS records of a matched book (`ABS_EBOOK`, `ABS_AUDIO`) stay mutually consistent at a **shared `lastUpdate`**, because every writer — any device's live `CanonicalSyncCycle`, and the dual-write — writes **both** records from one canonical position at one timestamp. This is what makes the **single-target** worker safe: reconciling each dirty row independently, both comparisons see the same server timestamp and reach the same verdict (both push, or both pull), so the records never split-brain.

Divergence is possible only transiently and is bounded — either **partial connectivity** (one record's PATCH succeeds, the other stays dirty for the next sweep) or **prerequisites absent** (a matched book with no bundle degrades to single-store until reopen). Both are eventually-consistent and no worse than ADR 0019's existing per-target PATCH isolation, which already tolerates one record lagging. The worker therefore does **not** need to reconcile the pair atomically.

## Scope

Progress only (`reading_positions` + `audiobook_positions`). To Read toggles, mark-read, and annotation sync are **not** queued here (they already fail loudly with an optimistic-flip-plus-snackbar, so no silent loss); the dirty-set is left shaped to admit them later via a decoupled `sync_state` table if they join.

## Consequences

- A matched book read/listened offline ends up with **both** local rows dirty (the dual-write), so the worker pushes **both** ABS records on reconnect — the sibling record no longer stays stale until reopen, and the worker still does zero translation. The cost is one extra local write per genuine position change on matched books, plus the sibling-store dependency on each live surface.
- Two reconcile callers, **one** reconcile implementation — closes the drift risk ADR 0029 warned about and retro-fixes the ServerWins-persists-only-timestamp gap for the live reader.
- A concurrent offline edit during a worker round-trip is never lost or clobbered: compare-and-clear abandons the round's conclusion and the next sweep re-evaluates against current values.

## Alternatives considered

- **Event log of position changes (rejected).** Reading position is last-writer-wins on a single value; replaying intermediate page turns is pointless and complicates the conflict check. Coalesced dirty-set is the natural model.
- **Worker-only reconcile primitive, live paths unchanged (rejected).** Leaves two implementations to drift and the ServerWins-persist gap in the live path. The live loop must write `lastSyncedAt` for the dirty-flag coordination to work anyway, so it is already coupled — unifying the engine finishes that coupling honestly.
- **In-process connectivity callback instead of WorkManager (rejected).** Dies with the process, so it cannot cover the never-reopened / app-killed case — the whole point.
