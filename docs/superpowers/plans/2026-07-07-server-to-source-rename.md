# Server → Source Rename Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Issue:** [#432](https://github.com/pkmetski/riffle/issues/432) — foundation of the [ADR 0041](../../adr/0041-source-and-service-abstractions-replace-server.md) multi-source epic.

**Goal:** Rename the `Server` concept to `Source` across the Room schema, `core:database`, `core:domain`, `core:data`, and the `:app` module. Zero user-visible change; zero semantic data change.

**Architecture:** One Room migration (43 → 44) renames the `servers` table to `sources`, adds a `type TEXT NOT NULL DEFAULT 'ABS'` column, and renames `serverId` → `sourceId` on every carrying table. Existing `serverType` column stays as-is (Storyteller row extraction is deferred to #441 per Q1 decision). Kotlin renames are mechanical: `ServerEntity/Dao/Repository/FilesCleaner` → `SourceEntity/Dao/Repository/FilesCleaner`, `Server`/`ServerId` (String) → `Source`/`SourceId` (still String, no value class introduced), new `SourceType` enum with a single `ABS` entry. Storyteller-adjacent files (identified by having "Storyteller" in the path/filename) keep `Server` in their identifiers — they are renamed in #441.

**Tech Stack:** Kotlin, Room 2.6, Hilt, Gradle (KSP for Room schema export).

## Global Constraints

- Room `@Database` version bumps from 43 → 44. New migration is `MIGRATION_43_44`.
- Schema JSON must export to `core/database/schemas/com.riffle.core.database.RiffleDatabase/44.json` before the PR is opened (KSP writes it during a build).
- The new migration must be registered in `DataModule.kt`'s `addMigrations(...)` list.
- `MigrationTest.kt` gets a new `@Test fun migration43To44_renamesServersToSources()` AND the new migration is added to the `migrateFullChain` `runMigrationsAndValidate` call.
- Storyteller-adjacent files are OUT of scope — do NOT rename any file with "Storyteller" in the path or filename that still references `Server*`. Those are handled by #441.
- No user-visible copy changes. The Nav Drawer still says "Server" (that will change in #435).
- Every existing test keeps passing untouched. Only pure renames — no behaviour change. If a test needs to be renamed because it references `ServerRepository`, update the reference; do not touch its assertions.
- Existing `serverType` column stays (holds `AUDIOBOOKSHELF` / `STORYTELLER`). New `type` column is added and defaulted to `'ABS'` — the two coexist until #441.
- `SourceId` and `SourceItemId` remain raw `String` types (no value class). The domain already uses raw `String`; do not introduce a wrapper.
- Concurrent Gradle KSP corruption is a known issue in this repo — run JVM suites sequentially, not in parallel.

Rationale: write the migration SQL and its test first, at a schema level, before touching the Kotlin entities. This isolates the schema change from the entity/DAO rename churn.

**Migration SQL strategy (Room):**
Room migrations can rename tables via `ALTER TABLE ... RENAME TO ...`, but cannot rename columns pre-SQLite 3.25 in a way Room's schema validator likes reliably. **Use the recreate pattern** for every carrying table:

1. `ALTER TABLE servers RENAME TO sources_old;`
2. Create the new `sources` table with `type TEXT NOT NULL DEFAULT 'ABS'` at the end.
3. `INSERT INTO sources (id, url, isActive, insecureConnectionAllowed, username, serverType, absUserId, type) SELECT id, url, isActive, insecureConnectionAllowed, username, serverType, absUserId, 'ABS' FROM sources_old;`
4. `DROP TABLE sources_old;`
5. For each carrying table (13 primary + join tables + `libraries`):
   - Create a `<name>_new` table with `sourceId` in place of `serverId`, keeping all other columns, indices, and FKs (retargeted to `sources(id)`).
   - `INSERT INTO <name>_new (sourceId, <rest>) SELECT serverId, <rest> FROM <name>;`
   - `DROP TABLE <name>;`
   - `ALTER TABLE <name>_new RENAME TO <name>;`
   - Re-create indices that Room expects on the renamed columns.
6. For `readaloud_links`, `readaloud_candidates`, `readaloud_dismissals`: rename `absServerId` → `absSourceId` and `storytellerServerId` → `storytellerSourceId` in the same recreate step.

**Interfaces:**
- Consumes: nothing.
- Produces: `MIGRATION_43_44: Migration` referenced by name in later tasks.

    }
